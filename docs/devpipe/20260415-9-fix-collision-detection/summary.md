## 原始需求

**来源**：GitHub Issue [#9](https://github.com/hexueyuan/index/issues/9)

**原始描述**：

> 现有的碰撞检测有点过于严格，有的地方只有一格通道，我需要在这里反复多次尝试才能避免碰撞走进通道里。

**业务背景**：本项目是一个基于 Phaser 3 的 Pokemon 风格 2D 俯视角游戏。地图中存在若干 1 格宽（16px）的通道，玩家在尝试通过这些通道时体验很差，需要反复调整位置才能进入，严重影响游戏流畅度。

## 需求分析过程

### 问题定位

通过代码分析，发现问题的根本原因是**玩家碰撞体与瓦片等大**：

- 地图瓦片尺寸：16x16 px
- 玩家碰撞体：16x16 px（Phaser Arcade 默认使用 sprite 全尺寸作为 AABB）
- 1 格宽通道：恰好 16px

这意味着玩家碰撞体和通道宽度完全相等，**零容差**。Phaser Arcade 物理引擎使用浮点数坐标，任何亚像素级别的偏移都会导致碰撞判定触发，实际上几乎不可能对齐通过。

### 影响面分析

碰撞体定义在 `Player` 类（`src/objects/Player.ts`）的构造函数中。`Player` 未调用 `setSize()` 或 `setOffset()`，使用了 Phaser 默认的全尺寸碰撞体。碰撞注册在 `MainScene.ts` 中通过 `physics.add.collider(player, layer)` 完成——这部分不需要修改。

### 方案选择

考虑了两个方案：

1. **仅缩小碰撞体**：将 16x16 缩小到 10x10，偏移到脚部区域，每侧 3px 容差
2. **缩小碰撞体 + 归一化斜向速度**：额外修复斜向移动速度为直线的 √2 倍的问题

最终选择方案 1，理由是：
- 碰撞体缩小是核心修复，3px 容差已足够顺畅通过 1 格通道
- 斜向速度问题影响较小，且属于独立的体验优化，可后续单独处理
- 最小化改动范围，降低引入新问题的风险

### 碰撞体参数设计

选择 10x10 而非其他尺寸的考量：
- **10px 宽**：在 16px 通道中每侧留 3px 容差，足够玩家自然通过
- **偏移 (3, 6)**：碰撞体居中水平放置（左右各 3px 边距），垂直方向偏下（顶部 6px 空间），模拟脚部碰撞区域。这是 Pokemon 类俯视角游戏的标准做法——角色的"存在感"在脚下而非头顶

## 实现方案

修改 `src/objects/Player.ts`，在构造函数的 `scene.physics.add.existing(this)` 之后添加碰撞体设置：

```typescript
const body = this.body as Phaser.Physics.Arcade.Body;
body.setSize(10, 10);
body.setOffset(3, 6);
```

碰撞体在 16x16 sprite 内的位置示意：

```
+----------------+
|                |  ← 6px 顶部空间（头部不参与碰撞）
|   +--------+   |
|   | 碰撞体 |   |  ← 10x10 有效碰撞区域
|   | 10x10  |   |
|   +--------+   |
+----------------+
 3px          3px   ← 左右各 3px 容差
```

改动量极小：1 个文件，8 行新增代码（含注释）。无新增依赖，无接口变更，完全向后兼容。

## 问题与解决方案

### 问题 1：SSH 推送失败

- **现象**：`git push` 报 `muxserver_listen: link mux listener ... Bad file descriptor`，无法推送到 GitHub
- **原因**：SSH 多路复用（ControlMaster）的 socket 文件状态异常
- **解决方案**：在 push 命令中禁用 SSH 多路复用：`GIT_SSH_COMMAND="ssh -o ControlMaster=no" git push origin HEAD:fix-collision-detection`

### 问题 2：容器内开发服务器外部无法访问

- **现象**：`npm run dev` 启动后，从宿主机无法访问 `localhost:5173`
- **原因**：Vite 默认绑定 `localhost`，在容器内运行时外部无法访问
- **解决方案**：添加 `--host 0.0.0.0` 参数：`npm run dev -- --host 0.0.0.0`

## 反思与复盘

### 做得好的地方

- **精准定位**：通过代码分析快速定位到碰撞体尺寸是根因，没有在地图数据或物理引擎配置上绕弯路
- **最小改动**：8 行代码解决问题，没有引入额外复杂度
- **参数设计有依据**：10x10 + offset(3,6) 的选择基于 Pokemon 类游戏的碰撞体设计惯例，而非随意调整

### 可以改进的地方

- 斜向移动速度未归一化（约为直线的 1.41 倍），虽然不在本次 Bugfix 范围内，但值得后续作为独立优化处理
- 碰撞体参数（10x10）是静态值，如果后续需要调整，可考虑提取为常量

### 后续建议

- 考虑在 `Player` 类中将碰撞体尺寸定义为命名常量，便于后续微调
- 如果地图设计中出现更窄的通道需求，当前 3px 容差可能需要重新评估
- 斜向速度归一化可作为独立 Issue 跟进
